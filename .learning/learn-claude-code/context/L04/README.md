### 1.2 专家视角

#### 专家共识

| 议题              | 共识内容                                                     |
| --------------- | -------------------------------------------------------- |
| **Subagent 本质** | Subagent 不是"更小的 Agent"，而是"干净上下文的委托执行"。核心价值是上下文隔离，不是性能优化。 |
| **返回值机制**       | 返回最后一次响应的 text 部分（LLM 的总结），不是 tool_use block。中间过程全部丢弃。   |
| **禁止递归**        | 这是 Harness 层的主动设计决策，不是技术限制。目的是防止目标偏移和成本失控。               |
| **创建成本**        | Subagent 创建成本是一次 API 调用，不是"小"。但换来的是父端上下文清洁。              |
| **扁平 vs 层级权衡**  | 扁平设计保持顶层控制，成本可控；层级设计允许层层委托但风险更大（目标偏移、成本失控）。              |

#### 专家分歧

| 议题 | 观点 A | 观点 B | 本课程立场 |
|------|--------|--------|------------|
| **扁平 vs 层级** | 扁平设计好：顶层保持控制，成本可控 | 层级设计好：层层委托更灵活 | 扁平（禁止递归），但承认层级设计有合理性 |

### 1.3 深度测试问题

> **Q1**: 为什么 Subagent 用独立 messages[] 而不是共享父端 messages？
>
> **思考引导**：
> - 考虑父端上下文的价值
> - 考虑子端可能执行的复杂任务
> - 考虑"丢弃中间过程"的意义
>
> **预期理解层级**：
> - L0（表面）：因为代码就是这样写的
> - L1（关联）：为了不污染父端上下文
> - L2（深层）：上下文隔离是认知保护——父 Agent 保持思维清晰，Subagent 的 30+ 次工具调用全部丢弃

> **Q2**: 为什么禁止递归（子端无 task 工具）？
>
> **思考引导**：
> - 考虑递归的目标偏移问题
> - 考虑成本预测
> - 考虑扁平 vs 层级权衡
>
> **预期理解层级**：
> - L0（表面）：因为 messages 会爆炸
> - L1（关联）：防止无限嵌套
> - L2（深层）：设计决策——扁平设计保持顶层控制，防止目标偏移和成本失控。不是 messages 问题，是架构选择。

> **Q3**: Subagent 返回什么？
>
> **思考引导**：
> - 考虑返回值的内容类型
> - 考虑中间过程的去向
> - 考虑父端收到的是什么
>
> **预期理解层级**：
> - L0（表面）：返回执行结果
> - L1（关联）：返回工具调用的结果
> - L2（深层）：返回最后一次响应的 text 部分（LLM 的总结），不是 tool_use block。中间过程全部丢弃。

## 五、费曼检验

> *"如果你不能简单地解释它，你就没有真正理解它。"* —— 费曼

### 5.1 简单解释 Subagent

**挑战**：用不超过 3 句话，向一个不懂编程的人解释 Subagent。

**我的解释**：
```
想象老板派一个员工出去办事。员工自己处理所有细节，读文件、打电话、跑腿，
但老板不需要知道员工具体读了哪些文件、打了哪些电话。
员工回来只说一句话："事情办好了，结果是这样。"老板继续工作，脑子里干净清爽。
```

**检验标准**：
- ✅ 解释了核心机制（委托执行 + 只汇报摘要）
- ✅ 用了老板-员工类比
- ✅ 通过追问检验（为什么老板不需要知道细节？因为细节太多会扰乱思维）

### 5.2 简单解释 上下文隔离

**挑战**：用不超过 3 句话，向一个不懂编程的人解释上下文隔离。

**我的解释**：
```
就像每个人都用自己的笔记本记事。老板用老板的笔记本，员工用员工的笔记本。
员工记了一堆详细笔记，但回来时只撕下最后一页"总结"给老板。
老板的笔记本永远干净，不会被员工的详细笔记污染。
```

**检验标准**：
- ✅ 解释了核心机制（独立笔记本 + 只保留摘要）
- ✅ 用了笔记本类比
- ✅ 通过追问检验（为什么不把员工笔记本直接给老板？因为老板不需要那些细节）

### 5.3 简单解释 禁止递归

**挑战**：用不超过 3 句话，向一个不懂编程的人解释禁止递归。

**我的解释**：
```
老板派员工出去办事，但员工不能再派自己的员工。
如果员工也能派人，一层层派下去，最后可能不知道谁在干什么，
而且老板也不知道要付多少人的工资。
```

**检验标准**：
- ✅ 解释了核心机制（只能派一层，不能再派）
- ✅ 用了层级委托类比
- ✅ 通过追问检验（为什么不能再派？防止目标跑偏和成本失控）

### 5.4 简单解释 返回值机制

**挑战**：用不超过 3 句话，向一个不懂编程的人解释返回值机制。

**我的解释**：
```
员工回来汇报时，只说一句总结，不把所有通话记录、文件复印件给老板。
老板只听到"结果是这样"，然后继续自己的工作。
那些中间过程——打了什么电话、读了什么文件——全扔了。
```

**检验标准**：
- ✅ 解释了核心机制（只返回摘要，丢弃过程）
- ✅ 用了汇报总结类比
- ✅ 通过追问检验（为什么丢弃过程？因为老板只需要结果，不需要细节）

## 六、核心洞察记录

1. Subagent 不是"更小的 Agent"，是"干净上下文的委托执行"
2. 返回值机制：最后一次 text 部分，中间过程全部丢弃
3. 禁止递归是设计决策，不是技术限制——防止目标偏移和成本失控
4. 创建成本是一次 API 调用，不是"小"
5. 扁平设计保持顶层控制，层级设计允许层层委托但风险更大
6. LLM 语义驱动需关键信息（有损压缩），Unix 数据驱动需完整数据（无损传输）

## 七、核心代码模式

```python
def run_subagent(prompt: str) -> str:
    sub_messages = [{"role": "user", "content": prompt}]  # 干净上下文
    for _ in range(30):  # safety_limit: 防无限循环兜底
        response = client.messages.create(
            model=MODEL, system=SUBAGENT_SYSTEM,
            messages=sub_messages,
            tools=CHILD_TOOLS,  # 禁止递归：子端无 task 工具
            max_tokens=8000,
        )
        sub_messages.append({"role": "assistant", "content": response.content})
        if response.stop_reason != "tool_use":
            break
        # 执行工具，append tool_result...
    return "".join(  # 返回最后一次 text 部分
        b.text for b in response.content if hasattr(b, "text")
    ) or "(no summary)"
```

**关键行解读**：
- 第 2 行：`sub_messages = [{"role": "user"}]` → 干干净净的独立 messages[]
- 第 3 行：`range(30)` → safety_limit 兜底，不是项目大小判断
- 第 7 行：`CHILD_TOOLS` → 子端无 task，禁止递归
- 第 10 行：`stop_reason != "tool_use"` → 循环退出条件
- 第 13-14 行：`return "".join(b.text...)` → 只返回 text 部分，丢弃 tool_use
