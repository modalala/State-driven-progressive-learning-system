# L09 Memory System 知识总结

## 一、核心概念

### 1.1 Memory的本质

| 层级 | 理解 |
|-----|-----|
| **L1** | Memory是跨对话的持久化，记录用户偏好、项目背景、反馈和外部引用。 |
| **L2** | Memory解决的是**长期认知积累**问题——让Agent不像第一次合作。设计原则：**存不可重新推导的信息**。 |

**核心定义**: Memory保存的是"以后还可能有价值、但当前代码里不容易直接重新看出来"的信息。

### 1.2 Memory vs Context Compact

| 维度 | Memory | Context Compact |
|-----|--------|-----------------|
| 目标 | 长期认知积累 | 短期上下文预算 |
| 生命周期 | 永久（跨对话） | 会话级（对话结束丢弃） |
| 存储位置 | System prompt | Messages[] |
| 是否被压缩 | ❌ 每轮新建 | ✅ tool_result被压缩 |

**设计张力**: 持久化问题用Memory，临时容量问题用Compact。

## 二、四种类型边界

### 2.1 类型定义

| 类型 | 存什么 | 判断法 |
|-----|--------|-------|
| **user** | 个人偏好（无法从代码推导） | "这是某一个人的偏好吗？" |
| **feedback** | 用户明确纠正或认可的行为 | "用户明确说过这样做对/不对吗？" |
| **project** | 团队约定、设计决策背后的原因（代码看不出"为什么"） | "这是团队约定吗？原因能从代码看出吗？" |
| **reference** | 外部资源位置（看板URL、监控地址） | "信息在项目外部吗？" |

### 2.2 类型边界案例

| 案例 | 正确类型 | 理由 |
|-----|---------|-----|
| "我喜欢简洁的回答" | user | 个人偏好 |
| "commit message必须包含ticket编号" | project | 团队约定（不是user） |
| "用Vue是因为合规要求" | project | 存的是**原因**（不是技术选型本身） |
| "bug跟踪在Linear项目INGEST" | reference | 外部资源指针 |
| "src/auth.py是认证入口" | ❌ 不存 | 文件路径可重新推导 |

### 2.3 Feedback的正负双向

**设计原则**: Feedback是"用户明确确认的行为模式"，无论正负。

- 只存纠错 → 系统越来越保守
- 不存认可 → 模型失去"正确行为"参考
- **类比**: 教育系统需要"正确答案"和"错误答案"双向反馈

## 三、不存清单

### 3.1 绝对不存的内容

| 内容 | 为什么不存 |
|-----|-----------|
| 文件结构、目录布局 | 可重新读代码得到 |
| 函数签名、模块路径 | 可重新读代码得到 |
| 当前任务进度 | 属于Task/Plan |
| 临时分支名、PR号 | 很快过时 |
| 修bug的具体代码 | 代码和git记录是准确信息 |
| 密钥、密码、凭证 | 安全风险 |
| 技术选型（Vue/React/MySQL） | 可从配置文件看出 |

### 3.2 判断原则

**核心原则**: 能从当前代码直接看出来就不存。

**追问法**: "这个信息能从代码直接看出来吗？能看出来就不存；看不出来就按类型存。"

### 3.3 用户要求存但不应存

即使用户说"帮我记住"，也要追问：
> "这里面真正值得长期留下的、非显然的信息到底是什么？"

例子：
- "记住当前PR编号" → ❌ 属于git历史，不存
- "记住今天改了哪些文件" → ❌ 属于commit记录，不存
- "记住用MySQL是因为合规不能用PostgreSQL" → ✅ 存**原因**（project类型）

## 四、存储结构设计

### 4.1 结构组成

```
.memory/
├── MEMORY.md          # 索引（最多200行）
├── prefer_tabs.md     # 单条memory（frontmatter + content）
├── feedback_tests.md
└── incident_board.md
```

### 4.2 Frontmatter格式

```markdown
---
name: prefer_tabs
description: User prefers tabs for indentation
type: user
---
The user explicitly prefers tabs over spaces...
```

### 4.3 为什么是"每条.md + 索引"

| 设计目标 | 实现方式 |
|---------|---------|
| 增量更新 | 新增memory只需写一个新文件，不重写大文件 |
| 快速遍历 | 索引200行快速定位，不需读所有文件 |
| 独立管理 | 删除/修改只需操作单个文件 |

**类比**: 类似数据库"索引+数据文件"分离，或博客"文章列表+单篇文章"结构。

### 4.4 memories字典结构

```python
memories = {
    "prefer_tabs": {
        "description": "...",  # 索引层信息
        "type": "user",        # 分类
        "content": "...",      # 正文层信息
        "file": "prefer_tabs.md"
    }
}
```

**设计逻辑**: name作为key便于查找，description用于索引显示，type用于分类组织，content用于模型理解。

## 五、生命周期与时机

### 5.1 Memory注入流程

```
会话开始
  |
  v
load_all(): 从磁盘加载memory文件 → memories字典
  |
  v
每轮agent_loop开头
  |
  v
build_system_prompt()
  | 
  +-- Part 1: agent描述
  +-- Part 2: memory_section (从memories字典生成)
  +-- Part 3: MEMORY_GUIDANCE
  |
  v
System prompt注入LLM调用
```

### 5.2 save_memory时机

```
save_memory工具调用
  |
  v 立刻执行（同步）：
  1. 写入.md文件（持久化）
  2. 更新memories字典（内存缓存）
  3. rebuild_index()（更新MEMORY.md）
  |
  v 返回tool_result
  |
  v 下一轮build_system_prompt() → 模型看到新memory
```

**关键**: save_memory是**同步更新**，不是异步"下一轮加载"。

### 5.3 System prompt vs Messages生命周期

| 位置 | 生命周期 | 是否被Compact |
|-----|---------|--------------|
| System prompt | 每轮新建 | ❌ 不受影响 |
| Messages[] | 累积追加 | ✅ tool_result被压缩 |

**类比**: System prompt像背景音乐（每轮重新播放），Messages像对话记录（累积但可能被压缩）。

## 六、漂移风险与处理

### 6.1 漂移的本质

Memory是"过去的时间点快照"，代码/项目是"当前动态状态"。

**漂移类型**:
- 文件名变了
- 团队约定改了
- 用户偏好变了
- 技术选型换了

### 6.2 防止原则

| 步骤 | 操作 |
|-----|-----|
| 1 | 把memory当作**方向提示**，不是绝对答案 |
| 2 | 使用前先读当前文件/配置验证 |
| 3 | 如果冲突，**优先相信刚观察到的真实状态** |
| 4 | 推荐具体路径前再验证一次 |

### 6.3 冲突处理案例

| 场景 | Memory说 | 代码实际 | 正确做法 |
|-----|---------|---------|---------|
| 数据库 | 用MySQL | PostgreSQL | 优先相信代码实际状态 |
| 文件入口 | auth.py | auth_service.py | 优先相信代码实际状态 |
| 团队约定 | commit带ticket | 不要求了 | 优先相信当前约定 |

**类比**: Memory像地图（可能过时），当前代码像实际地形。地图告诉你方向，走到那里要看实际路况。

### 6.4 用户说"忽略memory"

**设计原则**: 用户明确说"忽略"，就真的不参考。

实现方式:
- build_system_prompt()时不注入memory_section
- 或load_all()后清空memories字典

**边界**: 这轮忽略memory，下一轮如果用户没说，memory仍然可用（用户指令是临时的）。

## 七、Dream Consolidator机制

### 7.1 核心目的

防止memory变成垃圾堆——通过merge、delete、keep决策清理过时内容。

### 7.2 7 Gates检查

| Gate | 检查内容 | 防止什么问题 |
|-----|---------|-------------|
| 1 | enabled开关 | 用户主动禁用整合 |
| 2 | 目录存在+有文件 | 无数据时整合没意义 |
| 3 | 非plan模式 | 只读模式下不该修改memory |
| 4 | 24h冷却 | 过度整合消耗资源 |
| 5 | 10min间隔 | 频繁扫描浪费性能 |
| 6 | ≥5个session | 数据太少整合没意义 |
| 7 | 锁文件释放 | 多进程同时整合导致冲突 |

**Gate分类**:
- Gate 1-3: 功能开关（前置条件）
- Gate 4-6: 频率控制（资源保护）
- Gate 7: 并发安全（冲突防止）

### 7.3 4个Phase顺序

| Phase | 名称 | 作用 | 为什么这顺序 |
|-----|------|-----|-------------|
| 1 | Orient | 扫描MEMORY.md索引 | 先了解整体结构 |
| 2 | Gather | 读所有memory文件 | 获取完整内容做分析 |
| 3 | Consolidate | merge+delete+keep | 基于内容做决策 |
| 4 | Prune | 限制200行 | 最后确保索引不超限 |

**类比**: 类似房间整理——先看整体(Orient)，再拿东西(Gather)，分类整理(Consolidate)，最后扔掉多余的(Prune)。

### 7.4 Gate 6为什么需要5个session

**设计原则**: 整合需要足够样本量才能做出有意义决策。

- 只有1-2个session → memory数量很少
- 整合需要"merge + delete + keep"判断
- 数据太少无法判断哪些该合并、哪些该删除

**类比**: 类似统计需要足够样本——只有2条日志就做分析没有意义。

## 八、索引200行限制

### 8.1 设计意图

| 目的 | 说明 |
|-----|-----|
| 给人阅读 | 索引200行方便人快速定位 |
| 不是限制注入量 | 个体memory文件仍然存在 |
| load_memory_prompt不受影响 | 加载所有memory文件（不受索引截断限制） |

### 8.2 超过200个memory时

- 索引截断显示"... (truncated at 200 lines)"
- 个体memory文件仍然存在
- Dream Consolidator负责整合清理

**真正限制注入量的**: Dream Consolidator的整合机制，不是索引截断。

## 九、与其他机制的边界

### 9.1 Memory vs Task vs Plan vs CLAUDE.md

| 机制 | 生命周期 | 存什么 | 判断法 |
|-----|---------|--------|-------|
| Memory | 永久 | 跨会话有价值且不可推导 | "以后很多会话都可能有用吗？" |
| Task | 任务级 | 当前工作做什么+依赖 | "只对这次任务有用吗？" |
| Plan | 会话级 | 这一轮怎么做的过程 | "是临时安排吗？" |
| CLAUDE.md | 项目级 | 长期固定规则说明 | "是项目级稳定规则吗？" |

### 9.2 判断决策树

```
信息出现
  |
  v 问："只对这次任务有用吗？"
  |-- 是 → Task/Plan
  |-- 否 → 问："能从代码直接看出来吗？"
      |-- 是 → ❌ 不存
      |-- 否 → Memory
          |
          v 问："是项目级固定规则吗？"
          |-- 是 → CLAUDE.md
          |-- 否 → Memory（按类型存储）
```

## 十、设计原则与可迁移模式

### 10.1 核心设计原则

| 原则 | 说明 |
|-----|-----|
| 存不可推导 | 能从代码看出来就不存 |
| 边界清晰 | 四类型有明确判断法 |
| 同步更新 | save_memory立刻更新缓存 |
| 动态注入 | 每轮重建system prompt |
| 漂移防护 | 使用前验证当前状态 |
| 后台清理 | Dream Consolidator防止垃圾堆 |

### 10.2 可迁移模式

| 模式 | 可迁移场景 |
|-----|-----------|
| 四类型分类 | 任何Agent系统 |
| Gate检查模式 | 定期后台任务（备份、清理、刷新） |
| 索引+文件分离 | 增量存储系统 |
| 同步更新+动态注入 | 实时配置系统 |
| 漂移验证 | 任何依赖历史信息的系统 |

### 10.3 类比总结

| 概念 | 类比 |
|-----|-----|
| Memory | 地图（可能过时，给方向提示） |
| 当前代码 | 实际地形（真实状态） |
| Dream Consolidator | 房间整理（Orient→Gather→整理→扔掉） |
| System prompt | 背景音乐（每轮重新播放） |
| Messages | 对话记录（累积但可能被压缩） |
| Gate模式 | 后台任务防护（前置条件+频率限制+并发保护） |

## 十一、save_memory参数设计

| 参数 | 作用 | 为什么必需 |
|-----|------|------------|
| name | 标识符 | 系统查找、文件命名 |
| description | 一行摘要 | 快速定位、索引显示 |
| type | 分类 | 按类型组织、注入顺序 |
| content | 完整内容 | 模型理解具体信息 |

**设计逻辑**: description是"索引层信息"，content是"正文层信息"。分离是为了快速遍历+详细阅读的两层需求。

---

## 十二、memory_section注入顺序

**顺序**: `agent描述 → memory_section → MEMORY_GUIDANCE`

**为什么这样排序**:
- memory_section是"已记住的内容"（你记得什么）
- guidance是"记忆规则"（你应该记住什么）
- 先看已有内容，再看规则，逻辑顺序更清晰

**类比**: 先看菜单（已有菜品），再看烹饪规则（怎么做菜）。
