# L14: Cron Scheduler (定时调度)

## 核心概念

### 本质定位

> 定时调度解决的是"将来某个时间再开始做事"，而不是"稍后回来拿结果"。

**与 L13 Background Tasks 的区别**：

| 机制 | 回答的问题 | 关注点 |
|---|---|---|
| 后台任务 | "已启动的慢操作，结果什么时候回来？" | **等结果** |
| 定时调度 | "一件事应该在未来什么时候开始？" | **等开始** |

### 三组件心智模型

```
调度记录 → 定时检查器 → 通知队列
```

完整流程：
```
schedule_create(...)
  → 把记录写到列表或文件里
  → 后台检查器每分钟看一次"现在是否匹配"
  → 如果匹配，就把 prompt 放进通知队列
  → 主循环下一轮把它当成新的用户消息喂给模型
```

**关键洞察**：定时调度不是另一套 agent，最终还是回到同一条主循环。

## 关键数据结构

### ScheduleRecord

```python
schedule = {
    "id": "job_001",           # 唯一编号
    "cron": "0 9 * * 1",       # 定时规则（分 时 日 月 周）
    "prompt": "...",           # 到点后要注入主循环的提示
    "recurring": True,         # 是否反复触发
    "durable": True,           # 是否落盘保存（跨会话）
    "created_at": 1710000000,  # 创建时间
    "last_fired_at": None,     # 上次触发时间（防重复）
}
```

### 调度通知

```python
{
    "type": "scheduled_prompt",
    "schedule_id": "job_001",
    "prompt": "...",
}
```

---

## 核心字段解析

### last_fired_at

**作用**：防止短时间内重复触发同一任务。

没有这个字段，系统容易在检查周期内多次触发。

### durable

**作用**：是否落盘保存。

- `true`：程序重启后调度记录仍然存在，跨会话可用
- `false`：仅内存存储，重启后丢失

### recurring

**作用**：是否反复触发。

- `true`：周期性任务，每次触发后继续保留
- `false`：一次性任务，触发后删除

---

## 实现要点

### 检查周期

教学版建议**分钟级**检查，不是秒级。

原因：
1. 大多数 cron 任务本身不需要秒级精度
2. 秒级检查资源消耗高
3. 检查是后台独立循环，不是每轮对话触发

### 通知注入机制

时间到了 → 发通知到队列 → 主循环下一轮统一注入 messages

**为什么走主循环而不是独立执行**：
- 系统行为透明
- 模型统一决策
- 主循环结构不乱

---

## Cron 表达式（5字段版）

```
分 时 日 月 周
```

示例：
- `*/5 * * * *` → 每 5 分钟
- `0 9 * * 1` → 每周一 9 点
- `30 14 * * *` → 每天 14:30

---

## 初学者常见错误

1. **沉迷 cron 语法细节** → 主线是"调度记录如何回到主循环"
2. **没有 last_fired_at** → 重复触发问题
3. **只放内存不落盘** → 重启后丢失，不是真正调度
4. **后台默默执行** → 应该走通知队列让主循环决定
5. **追求毫秒级精度** → 更重要的是"有计划地触发"

---

## 系统整合

到了 L14，系统有两条外部事件输入：
- 后台任务完成通知
- 定时调度触发通知

**统一方式**：都走通知队列，在下一轮模型调用前统一注入。
